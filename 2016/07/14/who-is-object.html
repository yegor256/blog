What is an object? I‚Äôve done a little research, and this is what I‚Äôve found:

  • ‚ÄúObjects may contain data, in the form of fields, often known as attributes; and code, in the form of procedures, often known as methods‚Äù‚ÄîWikipedia at the time of writing.

What is common throughout all these definitions is the word ‚Äúcontains‚Äù (or ‚Äúholds,‚Äù ‚Äúconsists,‚Äù ‚Äúhas,‚Äù etc.). They all think that an object is a box with data. And this perspective is exactly what I‚Äôm strongly against.

If we look at how C++ or Java are implemented, such a definition of an object will sound technically correct. Indeed, for each object, Java Virtual Machine allocates a few bytes in memory in order to store object attributes there. Thus, we can technically say, in that language, that an object is an in-memory box with data.

Right, but this is just a corner case!

Let‚Äôs try to imagine another object-oriented language that doesn‚Äôt store object attributes in memory. Confused? Bear with me for a minute. Let‚Äôs say that in that language we define an object:

c {
  vin: v,
  engine: e
}

Here, vin and engine are attributes of object c (it‚Äôs a car; let‚Äôs forget about classes for now to focus strictly on objects). Thus, there is a simple object that has two attributes. The first one is car‚Äôs VIN, and the second one is its engine. The VIN is an object v, while the engine is e. To make it easier to understand, this is how a similar object would look in Java:

char[] v = {'W','D','B','H',...'7','2','8','8'}; // 17 chars
Engine e = new Engine();
Car c = new Car(v, e);

I‚Äôm not entirely sure about JVM, but in C++ such an object will take exactly 25 bytes in memory (assuming it‚Äôs 64-bit x86 architecture). The first 17 bytes will be taken by the array of chars and another 8 bytes by a pointer to the block in memory with object e. That‚Äôs how the C++ compiler understands objects and translates them to the x86 architecture. In C++, objects are just data structures with clearly defined allocation of data attributes.

In that example, attributes vin and engine are not equal: vin is ‚Äúdata,‚Äù while engine is a ‚Äúpointer‚Äù to another object. I intentionally made it this way in order to demonstrate that calling an object a box with data is possible only with vin. Only when the data are located right ‚Äúinside‚Äù the object can we say that the object is actually a box for the data. With engine, it isn‚Äôt really true because there is no data technically inside the object. Instead, there is a pointer to another object. If our object would only have an engine attribute, it would take just 8 bytes in memory, with none of them actually occupied by ‚Äúdata.‚Äù

Objects are no longer boxes with data; they know where the data is, but they don‚Äôt contain the data.

Now, let‚Äôs get back to our new pseudo language. Let‚Äôs imagine it treats objects very differently than C++‚Äîit doesn‚Äôt keep object attributes in memory at all. It doesn‚Äôt have pointers, and it doesn‚Äôt know anything about x86 architecture. It just knows somehow what attributes belong to an object.

Thus, in our language, objects are no longer boxes with data both technically and conceptually. They know where the data is, but they don‚Äôt contain the data. They represent the data, as well as other objects and entities. Indeed, the object c in our imaginary language represents two other objects: a VIN and an engine.

To summarize, we have to understand that even though a mechanical definition of an object is correct in most programming languages on the market at the moment, it is very incorrect conceptually because it treats an object as a box with data that are too visible to the outside world. That visibility provokes us to think procedurally and try to access that data as much as possible.

If we would think of an object as a representative of data instead of a container of them, we would not want to get a hold of data as soon as possible. We would understand that the data are far away and we can‚Äôt just easily touch them. We should communicate with an object‚Äîand how exactly it communicates with the data is not our concern.

I hope that in the near future, the market will introduce new object-oriented languages that won‚Äôt store objects as in-memory data structures, even technically.

An object is the equivalent of the quanta from which the universe is constructed
