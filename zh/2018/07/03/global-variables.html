    What's Wrong With Global Variables?

    The following text is a partial translation of the original English article, performed by ChatGPT (gpt-3.5-turbo) and this Jekyll plugin:

    只有懒惰的人们才没有写过有关全局变量（global variables）是邪恶的文章。这一说法始于1973年，当时W. Wulf等人声称“非局部变量是导致程序难以理解的主要因素”。自那时以来，有很多其他原因被提出来，以说服程序员停止使用全局变量。我想我已经阅读了全部的这些原因，但并没有找到最让我烦恼的那一个：可组合性。简而言之，全局变量使得代码在与原始作者预期不同的方式下难以或不可能进行组合。

    最近我在Ruby上为Zold编写了一个Web前端，使用的是Sinatra框架。根据他们的文档，下面是一个Web服务器的启动过程：

    App.start!

    这里start!是App类的一个静态方法，你必须将其声明为默认父类Sinatra::Base的子类。要告诉应用程序监听哪个TCP端口，你必须预先配置它。

    require 'sinatra/base'
class App < Sinatra::Base
  get '/' do
    'Hello, world!'
  end
end
App.set(:port, 8080)
App.start!

    如果您想启动两个Web服务器，该怎么办？对于测试来说，这可能是一个非常合理的要求。例如，由于Zold是一个分布式网络，有必要测试一些服务器之间的通信方式。我做不到！绝对没有办法。因为Sinatra的设计假设整个应用范围内只能存在一个服务器。

    这个真的可以修复吗？让我们来看看他们的代码。Sinatra::Base类本质上是一个单例，不应该有多个实例。当我们调用App.set(:port, 8080)时，值8080被保存到一个单一实例的属性中。数字8080在Sinatra::Base的所有方法中都可以使用，无论这些方法是从哪个实例调用的。

    我相信他们没有使用真正的Ruby全局变量，因为他们知道它们是不好的。为什么它们不好以及有哪些替代方案——这些问题他们并没有深入研究。

    从技术上讲，他们的设计是“全局范围”。 Sinatra :: Base 将整个应用程序视为其可见范围。无论是谁调用它，都可以看到一切，包括之前的调用中创建的内容以及之前实例化的对象中的内容。这个“类”就像一个巨大的全局变量包。

    每个全局变量都是这种问题制造者。当应用程序很小且测试覆盖率低时，全局变量可能不会造成问题。但是，应用程序越大，自动化测试场景越复杂，就越难以构建依赖于全局变量、单例或类变量的对象。

    我的建议？无论如何都不要考虑使用任何全局变量。
