On the other hand, XML with XSLT implements perfectly the idea of separating data and presentation. XSL is a much more powerful language. That‚Äôs why it‚Äôs possible to avoid any formatting inside XML.

Using S3 is very simple. You create a ‚Äúbucket‚Äù with a unique name, upload your ‚Äúobject‚Äù into the bucket through their web interface or through RESTful API, and then download it again (either through HTTP or the API.)

Amazon ships the Java SDK that wraps their RESTful API. However, this SDK is not object-oriented at all. It is purely imperative and procedural‚Äîit just mirrors the API.

For example, in order to download an existing object doc.txt from bucket test-1, you have to do something like this:

  new GetObjectRequest("test-1", "doc.txt")
);
InputStream input = obj.getObjectContent();
String content = IOUtils.toString(input, "UTF-8");

Region region = new Region.Simple(key, secret);
Bucket bucket = region.bucket("test-1");
Ocket ocket = bucket.ocket("doc.txt");
String content = new Ocket.Text(ocket).read();

Atomic Counters at Stateful.co

  • 18 May 2014
  • 509 words
  • 2 minutes to read
  • comments

Amazon DynamoDB is a great NoSQL cloud database. It is cheap, highly reliable and rather powerful. I‚Äôm using it in many web systems.

There is one feature that it lacks, though‚Äîauto-increment attributes.

Say that you have a table with a list of messages:

+------+----------------------------+
| id   | Attributes                 |
+------+----------------------------+
| 205  | author="jeff", text="..."  |
| 206  | author="bob", text="..."   |
| 207  | author="alice", text="..." |
+------+----------------------------+

Every time you add a new item to the table, a new value of id has to be set. And this has to be done with concurrency in mind. SQL databases like PostgreSQL, Oracle, MySQL and others support auto-increment features. When you add a new record to the table, the value of the primary key is omitted and the server retrieves the next one automatically. If a number of INSERT requests arrive at the same time the server guarantees that the numbers won‚Äôt be duplicated.

However, DynamoDB doesn‚Äôt have this feature. Instead, DynamoDB has Atomic Counters and Conditional Updates, which are very similar features. Still, they‚Äôre not exactly the same.

In case of an atomic counter, you should create a supplementary table and keep the latest value of id in it.

In case of conditional updates, you should retry a few times in case of collisions.

To make life easier in a few of my applications, I created a simple web service‚Äîstateful.co. It provides a simple atomic counter feature through its RESTful API.
